NOTES · Rationale
  • Day N of 30 is a horizon, not a deadline. The pace indicator (● ON PACE mint / ◐ BEHIND amber) reads gates-cleared against days-elapsed — never shows red.

NOTES · Rationale
  • Required fields carry an amber * — the alarm color's lightest weight, attention not blocking.
  • Money is captured and stored as integer cents. The field takes cents; the live preview renders the formatted figure so the operator can sanity-check 150000 CENTS = A$1,500.00.
  • Currency selection re-symbols every money preview on the page — one source of truth for the tenant's display format.
  • Timezone is load-bearing: it defines the night boundary the whole console pivots on. Called out in the hint, not buried.
  • Take-rate is deliberately absent from provisioning — ⚑ placeholder, venue reporting only. No rate marketing on an operator surface.
  • Inputs are SURFACE-2; mono for structured values (timezone, currency, cents), sans for free text (venue name).

NOTES · Rationale
  • Border style is the state — read it before any label.
  • Solid mint border = LIVE. Carries a recency stamp (SYNCED 2m AGO) so the operator trusts the data is fresh, not just connected once.
  • Dashed border = STUB · SEEDED DATA. The surface works from day zero on demo data; a cyan CONNECT action swaps seed for real. Cyan = the forward, opt-in action.
  • Amber solid = AUTH EXPIRED. Alarm's tinted-fill weight — the token lapsed, action needed, but never red. RECONNECT is the recovery path.
  • Seeded stubs mean a tenant is never blocked by an unconnected source — they go live on seed and upgrade in place.
  • No connector ever shows red — expired auth is amber, the strongest weight used here.
  • Klaviyo setup = create one flow per Atlas metric. Delivery is server-side events, never a list blast — Atlas fires a metric event per consented guest and your Klaviyo flow sends the message. Create a flow triggered on each metric you want live: Atlas Event Match, Atlas Regulars Lock-In, Atlas Winback, Atlas Crew Rebook, Atlas Loyalty Claim, Atlas Campaign. A metric with no flow simply does nothing — opt-in per message type. Profiles match on email / phone / external_id (the Atlas guest id).

Grants & tenantassign scopes · edit tenant ●ALWAYS tenant:admin
3 DO scopes granted · SEE is universal
W3b · CONFIRM GRANT · WORKED EXAMPLE
This grant lets a Floor Manager, on the night: ● SEE every booking, table, tab and door state for the venue — always, no toggle.
● DO seat a party, block a table, confirm or cancel a held booking bookings:write, and check a guest in at the door door:checkin.
⊘ CANNOT refund a deposit, re-request a failed split-pay share payments:write, run closeout closeout:run, or reassign anyone's grants tenant:admin — those toggles are off.
GRANTED SCOPES · bookings:write  ·  floor:manage  ·  door:checkin
NOTES · Rationale
  • SEE is always-on, always mint. Every operator sees the whole venue — visibility is never rationed. Rationing sight breaks the shared-truth model the console runs on.
  • DO is what's granted. Each DO toggle maps 1:1 to exactly one scope string — no bundles, no hidden implied permissions.
  • Scope strings show on hover only (bookings:write). The table stays human-readable; the raw scope is there for the admin who needs it, out of the way for the one who doesn't.
  • W3b confirms in plain language with a worked example — what this person can and can't do on the night, not an abstract scope list. The ⊘ CANNOT lines matter as much as the grants.
  • Toggle on = cyan (the enabling action); off = neutral gray. Never red for an off permission.

NOTES · Rationale
  • Three lists, one glance. FLOWS IN (mint), FLOWS OUT (electric), and — the one that builds trust — NEVER FLOWS (amber, tinted panel).
  • NEVER FLOWS is the point of this screen. Stating the boundaries explicitly — no guest PII to promoters, no cross-tenant graph, no take-rate to guests — is what makes the contract worth accepting.
  • GO LIVE is gated, not hidden. It renders disabled with a ⊘ glyph and an amber gate-note naming exactly what's outstanding. The operator sees the finish line and the two steps left.
  • Prerequisites use the same status glyphs as W0 — ● cleared, △ outstanding — so the gate reads consistently with the checklist.
  • Accepting the contract (demo toggle) clears the last prereq and the button un-gates to a live mint GO LIVE — the one moment mint means "commit".
